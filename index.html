Learning Outcomes

Upon completion, you will be able to:

1. Specify the programming language and tools
Determine inspection tools, select files/modules to modify, recognize the language, choose the right editor.

2. Define the logical order of the code
Read code by language type, document logical arrangement with flowcharts, write comments from flowcharts.

3. Modify the code
Determine type, place, and manner of modification; add/modify code; document changes; execute according to instructions.

4. Test the code after modification
Review test plan, test the modified code, address programming errors if found.

Version control systems ‚Äî introduction with examples

VCS manage changes to source code over time. Example: Linux kernel uses Git; thousands of developers contribute simultaneously.

  • Multiple developers collaborate without overwriting each other.
  • History of changes; conflict resolution.
  • Commands: git init, git add, git commit, git log.

Tracking changes

Each commit has a timestamp, author, and message. Use commit history to trace when a bug was introduced and fix it.

Commands: git add . ‚Üí git commit -m "message" ‚Üí git log

Collaboration and branching

Branching creates isolated copies. Use main for stable releases and feature branches for new work.

Commands: git branch feature-name, git checkout feature-name, make changes, git merge feature-name.

❮11❯
Merging and reverting changes

Merging: When branches are complete, merge into main. Resolve conflicts with git mergetool or manual edit.

Reverting: git revert <commit> creates a new commit that undoes a previous one. Use if a commit causes a crash.

Distributed version control and backup

In Git (DVCS), every developer has a full copy of the repository. Work offline, then git push when back online.

If the central server crashes, any clone can restore the project. git clone <url> gives a complete copy.

Overview of GitHub and its features

GitHub is built around Git. Key features:

  • Repositories ‚Äî Public or private; create via web or git init + git remote add.
  • Branches and pull requests ‚Äî Propose, review, and merge changes.
  • Issues ‚Äî Track bugs, enhancements; assign and label.
  • Project boards ‚Äî Kanban (To Do, In Progress, Done).
  • GitHub Actions ‚Äî Automate build, test, deploy on push/PR.

Case study: Development of Linux and creation of Git

Linus Torvalds (1991) created Linux; announced on comp.os.minix. Open-source collaboration grew the kernel.

BitKeeper was used for VCS; when its free license was revoked (2005), Torvalds created Git in weeks. Git is distributed, fast, with easy branching and merging. GitHub (2008) popularized Git further.

Version control systems

Version control (e.g. Git) manages changes to source code over time.

  • Tracking changes ‚Äî Every commit has a timestamp, author, and message.
  • Branching ‚Äî Work on features or fixes without affecting the main codebase.
  • Merging ‚Äî Combine branches; resolve conflicts when the same code is changed.
  • Reverting ‚Äî Roll back to a previous state if a change causes problems.
  • Auditing ‚Äî Clear record of who changed what and when.
Git is distributed: each developer has a full copy of the repository and can work offline.

Unit 1: Importance of code inspection

Code inspection (code review) finds defects, improves quality, ensures standards. Benefits: early bug detection, improved code quality, consistency, security, reduced technical debt.

Types of inspection tools: Static analysis & linters

Static code analysis ‚Äî Analyzes code without running it: errors, security (SQL injection, XSS), metrics (complexity, duplication). Examples: SonarQube, Coverity, Checkmarx, Fortify.

Linters ‚Äî Style, syntax, potential errors. Examples: ESLint (JS/TS), Pylint/Flake8 (Python), Checkstyle (Java), Rubocop (Ruby).

Managing pull requests

Procedures for creating and managing pull requests:

  • Create a branch for your modification; make commits with clear messages.
  • Push the branch and open a pull request with a title and description.
  • Respond to review comments; update code if requested.
  • After approval, merge (or squash) into the target branch.
Linking PRs to issues (e.g. "Fixes #123") keeps traceability and closes issues automatically when merged.
